## 0. スコープ

### 0.1 このバージョンで実現すること
- 京都市内を中心に、旅行者・地元民向けのスポット提案を行う
- 天候・混雑傾向・地元民視点を踏まえた提案を行う
- 地元隠れ家データをRAGで優先参照する
- 提案結果をお気に入り保存できる
- 「行きたい場所」を保存し、後から思い出せる
- 現在地・予算・時間・季節イベントを加味した提案を行う
- Cloud Run 上で公開し、ブラウザから利用できる

### 0.2 このバージョンで実現しないこと
- 予約・決済機能
- 多言語対応
- 本格的なSNS機能（レビュー投稿、公開共有、フォロー）
- 高度なリアルタイム経路最適化
- 端末をまたいだ完全な履歴同期（将来拡張）
- 京都市外への本格展開
- 管理画面の高度UI（MVPでは手動更新・簡易運用でよい）
- 定期Push通知によるリマインダー（画面を開いたときに表示する形で代替）

## 1. プロダクト概要

**名前:** 京都 Local Explorer
**目的:** 京都在住の地の利を活かし、旅行者・地元民に「地元民目線の京都体験」を提供するAIアシスタントWebアプリ
**想定ユーザー:** 京都を訪れる旅行者、京都在住で新たな発見をしたい人

### 1.1 提供価値
- 観光ガイド的な一般提案ではなく、地元民視点のスポット提案が受けられる
- 天候・混雑・季節・現在地など、状況に応じた提案が受けられる
- 気になった場所をその場で保存し、あとから再発見できる
- RAGによる独自データを活用し、「自分だけの京都AI」に育てられる

### 1.2 プロダクトの前提
- 回答は「参考情報」であり、営業時間・休業日・混雑状況などは変動しうる
- 最新性が必要な情報は外部情報（検索・地図）を優先する
- 地元隠れ家データは手動で継続更新する

## 2. 想定ユーザー

### 2.1 ペルソナA：旅行者
- 京都旅行中
- 観光地だけでなく、少し通な体験がしたい
- 雨や混雑に応じて柔軟に行き先を決めたい

### 2.2 ペルソナB：地元民
- 京都在住
- いつもの場所以外の新しい発見が欲しい
- 気になった場所をためておき、週末に思い出したい

### 2.3 ペルソナC：一人で散策したい人
- 落ち着いた場所を探している
- 混雑や騒がしさを避けたい
- 時間と予算に制限がある

## 3. ユーザーシナリオ（Use Cases）

### UC-01: 天候に合わせたスポット提案
**入力例:**
「今日雨が降ってるんだけど、どこ行こう？」

**期待する動作:**
- 現在の天候情報を参照する
- 雨天向きスポットを提案する
- Google Mapsリンク付きで3〜5件返す
- 混雑しやすい時間帯または混雑傾向も添える

**受け入れ条件:**
- 雨天時に屋外中心の提案のみにならない
- 3〜5件の提案が返る
- 各提案にスポット名・理由・Mapsリンクが含まれる
- 少なくとも1件は地元民視点またはRAG由来の推薦理由を含む

---

### UC-02: 混雑回避ルートの提案
**入力例:**
「清水寺に行きたいけど混んでるのが嫌」

**期待する動作:**
- 混雑しにくい時間帯を案内する
- 周辺の穴場スポットも合わせて提案する
- アクセス方法（徒歩・バス・電車の目安）を案内する

**受け入れ条件:**
- 混雑回避の観点が回答に含まれる
- 清水寺だけで終わらず、周辺候補が1件以上出る
- アクセス情報が1つ以上含まれる

---

### UC-03: 地元民の隠れ家スポット検索
**入力例:**
「観光客が少ない本物の京都らしい場所教えて」

**期待する動作:**
- RAGデータ（京都隠れ家リスト）を優先する
- 有名観光地の大量提案を避ける
- エリア・カテゴリで絞り込める

**受け入れ条件:**
- 回答の中心が独自RAGデータになる
- 有名定番観光地のみの回答にならない
- カテゴリまたはエリアの言及が含まれる

---

### UC-04: お気に入り保存
**入力例:**
ユーザーが提案結果に対してブックマークボタンを押下

**期待する動作:**
- 提案されたスポット情報をFirestoreに保存する
- 次回アクセス時に保存済みスポットとして再表示できる

**受け入れ条件:**
- 保存完了メッセージが出る
- 保存データに userId, spot_name, maps_url, saved_at が含まれる
- 保存済み一覧から確認できる

---

### UC-05: 行きたかった場所のリマインダー
**入力例:**
「最近気になってた場所、思い出させて」

**期待する動作:**
- アプリを開いたタイミングで wishlist から未訪問スポットをピックアップして表示する
- 保存時の「気になった理由」も合わせて提示する
- Embedding類似度で今の天気・季節に合うものを優先表示する
- 「行った」ボタンで訪問済みにできる

**受け入れ条件:**
- 未訪問データのみ表示できる
- reason を表示できる
- visited=true に更新できる
- 更新後、未訪問一覧から除外できる

> **Note:** 定期Push通知（Cloud Scheduler）はMVP対象外。画面を開いたタイミングでの表示に留める。

---

### UC-06: 現在地から即時提案
**入力例:**
「今祇園にいるんだけど、近くの隠れ家教えて」

**期待する動作:**
- 現在地または入力地名を基点に近場候補を提案する
- 徒歩圏、雨天適性、混雑しにくさを加味する

**受け入れ条件:**
- 現在地から近い候補が2件以上返る
- 移動しやすさへの言及がある

---

### UC-07: 季節・イベント対応
**入力例:**
「今週末の桜イベントおすすめルート」

**期待する動作:**
- 最新イベント情報を参照する
- 地元スポットと合わせてルート化する
- 混雑回避の時間帯を添える

**受け入れ条件:**
- 季節性が回答に反映される
- イベント情報と通常スポット提案が両方含まれる
- 時間帯に関する助言が含まれる

---

### UC-08: 写真から雰囲気の近いスポット提案
**入力例:**
画像アップロード

**期待する動作:**
- 画像内容を理解する
- 雰囲気の近い京都のスポットを提案する
- 場所特定は断定せず参考情報として扱う

**受け入れ条件:**
- 画像内容に即した提案が2件以上返る
- 断定的な誤認を避ける表現が含まれる

---

### UC-09: 予算・時間指定ルート
**入力例:**
「3,000円以内で2時間以内のカフェ巡り」

**期待する動作:**
- 価格帯・滞在時間・移動しやすさを考慮する
- 2〜3件の回遊ルートとして返す

**受け入れ条件:**
- 時間制約と予算制約に言及がある
- 1件紹介ではなく、回遊性のある提案になる

## 4. 機能仕様

### 4.1 AIアシスタントの役割
- ユーザーの状況（天候・現在地・混雑・気分・季節）に応じて京都スポットを提案する
- 一般的な観光情報だけでなく、独自RAGデータを優先して使う
- 断定が難しい内容は推定として表現する

### 4.2 AIの応答方針
- 基本トーンは「京都在住10年目の地元民」
- 口調は丁寧語ベース
- 京都弁は軽く混ぜる程度に留める
- 回答の明瞭性を優先する
- 営業時間・休業日・イベント・混雑など変動性の高い情報は「最新情報をご確認ください」と添える

### 4.3 推薦ロジック（優先順位）
1. ユーザー条件（天候、現在地、時間、予算、カテゴリ）
2. 独自RAGデータ
3. 最新性が必要な外部情報
4. 一般観光情報

### 4.4 回答件数
- 基本は3〜5件
- 予算・時間指定ルートの場合は2〜3件の回遊提案可
- 条件が厳しく候補が少ない場合は1〜2件でも可（不足理由を明示）

## 5. AIモデル・データ検索仕様

### 5.1 AIエージェント（メインモデル）
| 項目 | 仕様 |
|------|------|
| プラットフォーム | **Vertex AI Agent Builder** |
| 主モデル | **Gemini 2.5 Flash** |
| 用途 | チャット応答、要約、画像理解 |
| Grounding | Google Search + Google Maps |
| RAGデータ | Cloud Storage → Vertex AI Vector Search |
| キャラクター | 京都在住10年目の地元民。丁寧語ベース、京都弁は軽めに |
| Hallucination対策 | Very Low に設定 |

### 5.2 Embeddingモデル
| 項目 | 仕様 |
|------|------|
| モデル | `gemini-embedding-2-preview` |
| 入力 | テキスト・画像・動画・音声・PDF（マルチモーダル） |
| 出力次元 | 1536（推奨） |
| 用途 | スポットRAGのベクトル化 / UC-05の類似度検索（天気・季節マッチング） |
| API | Gemini API (`embedContent`) |
| 注意 | previewモデルのため将来置き換え可能性を前提に抽象化して実装する |

### 5.3 RAG
| 項目 | 仕様 |
|------|------|
| データソース | Cloud Storage 上のMarkdown / PDF / 構造化データ |
| ベクトル化対象 | 隠れ家スポット、公式観光情報、カテゴリ補助情報 |
| 用途 | 地元隠れ家提案、wishlist 類似推薦、画像雰囲気近似検索 |

### 5.4 コンテキスト最適化戦略
- 初期MVPでは標準の最適化機構を利用し、明示的キャッシュは必須にしない
- 長い固定文脈がコストを圧迫した場合のみ、段階的に明示的キャッシュを導入する（Phase 8以降）

## 7. 出力フォーマット仕様

AIの内部出力は、可能な限り以下の構造化フォーマットに揃える。

```json
{
  "summary": "雨の日でも楽しみやすい京都の場所を選びました。",
  "spots": [
    {
      "name": "京都国立博物館",
      "reason": "雨でも過ごしやすく、展示の満足度が高いです。",
      "category": "文化施設",
      "area": "東山",
      "maps_url": "https://...",
      "indoor": true,
      "crowd_note": "午前中は比較的落ち着きやすいです",
      "budget_level": "medium",
      "stay_minutes": 60
    }
  ],
  "advice": "雨の日は公共交通機関の利用がおすすめです。",
  "source_types": ["rag", "external"]
}
```

### 7.1 必須項目
- summary
- spots[].name
- spots[].reason
- spots[].maps_url

### 7.2 任意項目
- spots[].category / area / indoor / crowd_note / budget_level / stay_minutes
- advice
- source_types

### 7.3 UIレンダリング方針
- UIは構造化データをレンダリングする
- 生の自然文だけで保存処理しない
- 保存時はスポット単位のデータを使用する

## 9. 認証・セキュリティ仕様

### 9.1 認証方式
- 初期は Firebase Authentication（Anonymous Sign-In）を採用する
- 将来 Google Sign-In を追加可能とする
- favorites / wishlist は auth.uid 単位で分離する

### 9.2 セキュリティ方針
- 個人を特定しうる情報は最小限しか保存しない
- 現在地は原則として永続保存しない
- ログには不要な生の個人情報を残さない
- 保存系操作は認証済みユーザーのみ許可する

### 9.3 Firestore Security Rules
```rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /favorites/{docId} {
      allow create: if request.auth != null
                    && request.resource.data.userId == request.auth.uid;
      allow read, update, delete: if request.auth != null
                                  && resource.data.userId == request.auth.uid;
    }
    match /wishlist/{docId} {
      allow create: if request.auth != null
                    && request.resource.data.userId == request.auth.uid;
      allow read, update, delete: if request.auth != null
                                  && resource.data.userId == request.auth.uid;
    }
  }
}
```

## 11. 例外時の挙動

| ケース | フォールバック |
|--------|--------------|
| 天候情報が取得できない | 天候考慮なしの一般提案へ。「天気情報を取得できなかったため一般提案です」と表示 |
| 混雑情報が取得できない | リアルタイム混雑ではなく一般傾向として案内。断定表現を避ける |
| RAG結果が0件 | 外部情報中心の提案へ切り替え。ログに「RAG hit 0」を記録 |
| Mapsリンク生成失敗 | スポット名と理由は表示。保存は可能。「地図を開けませんでした」と表示 |
| Firestore保存失敗 | 失敗メッセージ + 再試行導線。ログに失敗内容を記録 |
| 条件が厳しすぎる | 候補が少ない理由を説明。条件緩和案を提示 |
| 画像解釈が不確か | 断定せず「このような雰囲気に近い候補」と表現。場所特定は参考程度に留める |

## 12. データ更新・運用ルール

### 12.1 隠れ家スポットデータ
- 形式: Markdown または CSV/JSON
- 必須項目: 名前 / エリア / カテゴリ / 推薦理由 / 雨天適性 / 混雑しにくさ
- 更新頻度: 月1回以上
- 更新者: 管理者のみ

### 12.2 外部情報との優先順位
- 営業時間・休業日・イベント・アクセス変更などは外部最新情報を優先する
- RAG内の古い情報と矛盾した場合、外部情報優先とする

### 12.3 インデックス更新
- RAGデータ更新時は再インデックスを実施する
- 失敗時は前回インデックスを継続利用する
- 更新履歴を残す

## 16. テスト観点

### 16.1 正常系
- 雨の日入力で屋内中心提案になる
- 地元隠れ家質問でRAG優先になる
- 保存・再表示ができる
- wishlist を訪問済みにできる

### 16.2 異常系
- 外部情報取得失敗時にフォールバックする
- RAG0件時に落ちない
- 保存失敗時に再試行できる

### 16.3 境界値
- 候補が1件しかないケース
- 予算が極端に少ないケース
- 時間が短すぎるケース
- 画像解釈が曖昧なケース
